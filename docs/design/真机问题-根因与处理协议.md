# 真机三症状：根因定位 + 问题清单处理协议（2026-07-10）

> 用户〖2026-07-10 真机实测〗：「有些角色图片是黑的」「打的时候可能是忍者龟带来的莫名卡顿」「打完在一半突然黑屏然后闪退了」
> 用户〖2026-07-10〗：「准备好处理我的问题清单，确保不要产生分歧，思考怎么做到」

## 一、根因（有数字，不是猜）

### ★ 角色立绘是**超长横条图**，宽度超过手机 GPU 的最大纹理边长

战斗单位的 idle 动画是 `pets/<id>.png`，把 N 个 500×500 的帧**横着排成一条**（`hframes = 宽 / 高`，`RealtimeBattle3DScene.gd:1744`）：

| 文件 | 尺寸 | 帧数 |
|---|---|---|
| `shell.png` | **10000×500** | 20 |
| `fortune.png` | **9000×500** | 18 |
| `headless.png` | **8500×500** | 17 |
| `hunter.png` | 7500×500 | 15 |
| `hiding.png` / `line.png` | 7000×500 | 14 |
| `space.png` | 6000×500 | 12 |
| `crystal.png` | 5500×500 | 11 |
| `candy.png` / `stone.png` | 5000×500 | 10 |

OpenGL ES 3 的 `GL_MAX_TEXTURE_SIZE`：**规范下限 2048**，真机常见 **4096 / 8192**。
超过上限的纹理**创建失败 → 采样出来就是黑的**。

→ **这就是「有些角色图片是黑的」**：宽度越大的越先黑（龟壳/财神/无头/幽灵 先黑；4096 上限的机器上，一大半龟都会黑）。
→ 桌面 NVIDIA 上限 32768，所以**端游一切正常**，我之前所有 headless/桌面测试都碰不到。

### ★ 所有贴图都是**未压缩** RGBA8

661 张 PNG，`compress/mode=0`（无损）→ 全量展开 **945.1 MB 显存**。
一场战斗要同时驻留 6 只龟的长条图（单张 13~20 MB）+ VFX。
在手机上撑爆显存 → 先黑图、再黑屏、最后被系统杀掉。**这就是「黑屏然后闪退」。**

### 「忍者卡顿」——**数据不支持**是忍者的问题
我用 `tests/probe_leak.gd` 量了 90 秒战斗的节点数/对象数：
- 忍者受审场：节点 128 → 117（**不涨**），对象 1862 → 2328（**5.2 个/秒**）
- 非忍者场：对象增长同样是 **5.2 个/秒**
→ 忍者**没有**特殊增长。你说「感觉不是忍者的问题」是对的。
卡顿更可能来自**贴图上传/显存换页**（黑图的同一个根因）。对象那 5.2/s 是 `_sim_tweens` 注册表攒到 512 才裁剪，**会平台化，退出时只漏 2 个音频对象**，不是泄漏。

### 端游 100 秒战斗有没有报错？（你问的）
- headless 120 秒：`SCRIPT ERROR = 0`、`Max recursion = 0`
- **窗口模式真 GL 渲染 100 秒**：`SCRIPT ERROR = 0`
- 两边都只有 **2 条 `ERROR: Infinite loop detected`**（某个 `set_loops()` 的 Tween 单圈时长为 0）——已列入待修，但它不会闪退。

### 顺带：25 组**零引用的重复副本**
`龟壳v1.png == shell.png`、`财神龟v1.png == fortune.png` … 内容逐字节相同，中文名那份**代码里 0 处引用**，却被打进 APK。

## 二、修法（分两类，避免"我替你决定"）

### A. 强制且无损，我直接做（不需要你拍板）
1. **删掉 25 组零引用的中文名副本**（先用带自检探针的脚本证明 0 引用）。
2. **把横条图重排成网格**（像素**逐字节不变**，只是换排布），使最大边 ≤ 4096：
   例：`shell.png` 20 帧 → 5列×4行 = 2500×2000。加载端已支持 `hframes`/`vframes`（`RealtimeBattle3DScene.gd:1514-1515`），只需补 `vframes` 推导。
3. 修 `set_loops()` 的零时长 Tween。

### B. 需要你拍板（**画质取舍，我不猜**）
| 决定 | 选项 | 影响 |
|---|---|---|
| 纹理压缩 | ① 开 VRAM 压缩(ETC2)　② 保持无损 | 开：显存 **÷4**，手机稳很多，但像素画可能有压缩噪点。不开：画质原样，低端机仍有 OOM 风险 |
| 最大纹理边长 | ① 4096（网格重排即可，像素不变）　② 2048（需要把帧从 500px 缩到 256px，**画质会降**）| 4096 覆盖绝大多数 ES3 真机；2048 才是规范下限，最保险但要缩图 |

我的建议：**先做 A（像素不变）+ 开 ETC2 只对 `pets/` 与 `vfx/`**，重出 APK 让你在真机上看一眼有没有噪点；不满意再退回无损。

## 三、你的问题清单——我怎么保证不产生分歧

每条反馈都走同一条流水线，**任何一步缺失就不动代码**：

1. **原话逐字入库**：把你写的那条原文照抄进 `docs/design/问题清单.md`，标日期。不改写、不概括。
2. **复现**：写一个能跑的探针/测试复现它（像这次的 `probe_leak.gd` / `smoke_scenes.gd`）。
   **复现不了就如实说"复现不了"**，不猜原因。
3. **证据**：贴出活代码行号 + 实测数字。结论只能来自「代码 + 实测」，不能来自我的记忆或我写的设计文档。
4. **归类**（决定谁来拍板）：
   - **事实错误**（实装 ≠ 你的原话 / 实装自相矛盾）→ **我直接修**，不问。
   - **取舍**（画质、手感、数值平衡、美术风格）→ **列成选项给你选**，我给建议但不替你定。
   - **我造成的**（我编的数值、我写岔的注释/文档）→ 单独标 ⛔，写清是我造成的。
5. **验证**：`bash run-tests.sh` ALL PASS + 存档 md5 不变 + 针对该问题的新测试。**写「已修」前必须真跑出来。**
6. **回填唯一知识库**：结论进 `28龟技能设计-权威.md`，每个值带出处等级（T1 你的原话 / T2 回合制原值 / T3 实装默认）。
7. **机器守卫**：能变成断言的就写进测试（如 `verify_codex_text.gd`），不靠"我这次很仔细"。

> 分歧的根源一直是同一个：**我拿自己写的文档当事实源，或只审我正在看的那个字段。**
> 所以协议里第 3 条写死「证据只能来自代码+实测」，第 7 条把结论钉进机器检查。
