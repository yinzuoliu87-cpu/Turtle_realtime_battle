## 关键架构事实（调查挖出的）
1. **duallane 整局结算当前零奖励**：打完每路 BattleScene 直接换场回 `DualLaneMapScene`（不走 `BattleEndScene`，那是 single/dungeon 的）。整局蛋碎 → `DualLaneMapScene._build_settlement()`(行440) 只显"胜/负"+回主菜单，**不给币、不动 meta coins、不调 record_battle_win**。→ 结算挂点 = **`DualLaneMapScene._build_settlement`（新建逻辑）**。
2. **背包/build 当前是局内临时、每局蒸发**：`equipped_p2`/`bench_inventory` 在 GameState 行40-86，`reset_dual_lane()`(行89) 每局清空。**"装备永不丢"的核心 = 把这俩从局内临时升级成 savegame 持久字段**——不是加个 save 那么简单。
3. **⚠ 阶段 2 ↔ 阶段 4 耦合**：结算给币公式 `25+2×现有心+5×已失心+(赢15)` 里的**心数 = 赛季/命系统(原阶段4)**；持久背包要写进**赛季存档(原阶段4)**。→ 持久化 + 结算 **依赖赛季/存档地基**。建议把"赛季/存档地基"提前到阶段 2 一起做（见下方推进建议）。
4. **小将无装备系统**：`_fill_lane_minions` 裸生成，无 equipped_p2 查询。背包"小将占位+装备"是阶段 3 大改造，阶段 2 schema 预留占位字段即可。
