# docs/plans/ — 需求方案书

**制度（用户 2026-07-22 定）：每个新需求，动手前先写方案书，写完留档。**

## 为什么

需求口述 → 直接开做，会出三类问题，都真实发生过：

1. **需求描述与代码事实不符，但没人先核对**
   2026-07-22 需求3 说「友军打不到被侵入的单位」，查代码发现方向是反的：
   原阵营友军**打得到**，打不到的是赛博自己那队。若直接照字面改，会改错方向。
2. **改动面被严重低估**
   同一天需求3 初版方案写「改 6 处」，穷举后发现真·敌我判定有 **20 处**。
   方案书强制"穷举 + 列行号"，这种低估会在动手前暴露。
3. **做完了才发现前提不成立**
   需求1 的 B 方案依赖「背景图左右边缘可横向外扩」——这个前提在方案阶段被读图确认，
   若留到实施阶段才发现不成立，整个方案要推倒。

## 放哪、怎么命名

```
docs/plans/YYYYMMDD-<需求简称>.md
```

写完**必须在 [docs/README.md](../README.md) 索引登记一行**，否则它会变成第 N 篇没人知道能不能信的文档。

## 必须包含的七节（缺一节就不算方案书）

| 节 | 内容 | 为什么不能省 |
|---|---|---|
| 1. 需求原文 | 用户**逐字**原话 | 复述会丢信息，也会把我的理解偷偷替换掉需求 |
| 2. 调查结论 | 现状 + **`文件:行号` 证据** | 没有行号的结论 = 推理，不算事实（见 CLAUDE.md §7） |
| 3. 需求 ↔ 代码的出入 | 哪几句和代码对不上，正确的事实是什么 | 这一节最值钱，见上面第 1 类问题 |
| 4. 方案 | 改哪些函数、为什么每处都要改、**穷举清单** | 防低估改动面 |
| 5. **已知风险与未决点** | 逐条列，标"已堵/未决/未验" | **不写这节就等于宣称"无漏洞"** |
| 6. 验收标准 | 怎么证明做完了，**且怎么证明这个检查会 FAIL** | 项目铁律：不能证伪的检查不算检查 |
| 7. 决策记录 | 用户拍板了什么、我建议了什么、分歧在哪 | 后来人不用重新推一遍 |

实施完成后**回填**：实际改了哪些、和方案的偏差、新发现的坑。
