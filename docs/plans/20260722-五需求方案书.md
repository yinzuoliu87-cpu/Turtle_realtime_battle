## 3. 调查结论（带行号）

### 3.1 选龟界面

- 场景 `scenes/TeamSelect.tscn` + `scripts/scenes/TeamSelectScene.gd`（UI 全程序化构建）
- 设计分辨率硬编码 **1647×955**（`TeamSelectScene.gd:19-20` `POC_W/POC_H`），不是 1280×720
- 缩放是自算舞台矩形做 contain（`:282-284 _stage_scale`），不是 Control 锚点
- `project.godot`：`viewport 1280×720` / `stretch mode=canvas_items` / `aspect=expand`

**实算（iPhone 横屏逻辑视口 1560×720，口径同 `tests/verify_ios_ui.gd:195`）：**

| 量 | 值 |
|---|---|
| contain 缩放 `s` | `min(1560/1647, 720/955)` = **0.7539** |
| 舞台 | 1242 × 720 |
| 左右留白 | **318 px，每边 159 px = 屏宽 20.4%**，全是纯色 `#241708` |
| 竖直余量 slack | **0.0 px** |
| `STAGE_OFFSET` 折算 | +46 px 下移（`:295-303` **无条件加，无 clamp**） |

→ 两条硬伤：**① 左右各 10% 死板纯色条 ② slack=0 却仍加正偏移 = 底边必被切**。

**附带发现**：`STAGE_ZOOM := 1.17`（`:264`）**全仓库只定义、零引用**（已 grep 全项目确认），是 cover→contain 那次改动的残留死代码，也是画布比旧版整整小 17% 的直接原因。

## 4. 方案

### 需求1 — B 方案（用户拍板）

背景 `select-bg` cover 外扩铺满屏，内容仍 contain，**两者共用同一 `s`**，只让背景在 cover/contain 差额区做边缘延伸。
> 红线（`TeamSelectScene.gd:279` 已记）：背景 cover / 内容 contain **各用各的缩放**会导致"背景上烤死的槽位框与内容对不上"。共用 `s` 是硬约束。

配套：`_stage_to_screen` 的 `off` 按 slack clamp（不切底）、删死常量 `STAGE_ZOOM`。

### 需求2 — 套已验证模板

三处（调试场选装备、调试场选龟、选龟龟池）改 `MOUSE_FILTER_PASS` + 位移阈值判点选；手机上关掉选龟卡片的 `set_drag_forwarding`，改"点卡片→点槽位"。

### 需求3 — 不改 `side`，新增统一原语（用户拍板）

```gdscript
func _is_hostile(a: Dictionary, b: Dictionary) -> bool:   # a 会不会攻击 b
    if is_same(a, b): return false
    if bool(b.get("hijacked", false)): return true                              # 被侵入者对所有人都是敌人
    if bool(a.get("hijacked", false)):
        return str(b.get("side","")) == str(a.get("_hijack_orig_side",""))      # 它只打原队友
    return str(a.get("side","")) != str(b.get("side",""))

func _is_ally(a, b) -> bool:   # 双向都不敌对才算友军 → 被侵入者是孤军
    return (not is_same(a, b)) and (not _is_hostile(a, b)) and (not _is_hostile(b, a))
```

**用用户的例子验算过（我方赛博+2友军，敌方仅 1 个被侵入单位）：**

| 主体 | 攻击目标 |
|---|---|
| 赛博龟 / 友军1 / 友军2 | 被侵入的敌人 ✅ |
| 被侵入的敌人 | **无目标 = 站着发呆** ✅ |

治疗/护盾：被侵入者**对谁都不是友军**（孤军）。存活数按原阵营算 → left=3 / right=1，双方都没被抹空。

**改动面：B 类 20 处全部收口 + `_sk_cyber_hijack` 去掉 `side` 改写 + `_acquire_target:4444` 的 `taunt_by` 分支加 `_is_hostile` 校验。**
`side` 全程不动 → 死亡不归队 / 跨路串阵营 / 单路瞬间判胜 / 召唤物串队 这四条 P1 **自动不存在**。

### 需求4 — 全量对齐两条路径（用户拍板）

抽公共 `_mitigate_and_land(u, dmg, dtype, raw, src)`，`_apply_damage` 与 `_apply_damage_from` 共用；诅咒改走正规路径并改成**每秒 tick**（不再 `dps*delta` 逐帧掉血）；废弃 `_raw_lose`。
**⚠ 这是数值平衡改动，不只是 bugfix：DOT 会明显变弱（钻石/岩石/虚化类单位尤其）。**

### 需求5 — 7 段动画（用户拍板 pro）

| # | 片段 | 对应时间 | 姿势 |
|---|---|---|---|
| ① | leap（蓄力+起跳） | 0.00–0.64 | 蹲身压低、三叉戟斜撑地 → 蹬地爆发向**后上方**腾起 |
| ② | throw（空中甩索） | 0.64–1.28 | 滞空抡臂蓄力 → 甩出绳索、身体前倾 → 绷紧待拉 |
| ③ | dive（俯冲伸脚） | 1.28–1.58 | **双脚前伸**、身体后倾拉紧绳索 |
| ④ | surf（踩滑，循环） | 1.58–2.41 | 重心压低、双臂张开平衡、三叉戟横持 |
| ⑤ | land（跳起落地） | 2.41 | 侧跳腾空 → 单膝落地缓冲 |
| ⑥ | run（走路，循环） | — | 三叉戟扛肩行进 |
| ⑦ | attack（普攻） | — | 三叉戟突刺 |

**档位**：①②③⑤⑦ 用 **pro**（5×20 = 100 额度）；④⑥ 两个**循环**动作用 **v3 指定 10 帧** —— 因为 pro 忽略 `frame_count` 固定 4 帧，4 帧循环会卡顿（上次精英实测）。
**配套**：扩 `_anim_key` 分流 front/back；把 `_elite_fix_norm` **泛化成按动画键查的归一表**，不再复制硬编码。

### 附加 — 远景背景第三版

**实拍证明的缺陷**：缩小 0.72 + 平移 9m 时画面上方 15% 是**纯黑**。算式：远景海床只铺 z∈[-30,-14]，最坏机位要看到 **z=-42.2**，**差 12.2 米**。三层"山脊"是 `billboard=DISABLED` 的**平板**。

最坏机位实算：

| 机位 | 画面上沿地面 z | 横向 x |
|---|---|---|
| 默认 zoom1 | -24.2 | ±31.5 |
| 缩小 0.72 | -33.2 | ±43.5 |
| 缩小+后移9m | **-42.2** | ±43.5 |
| 缩小+后移+侧移9m | **-42.2** | [-34.5, 52.5] |

改法：程序生成 **200×120 米真地形网格**（顶点噪声起伏）+ 山脊并进同一网格 + 地平线交给雾 + 保留鱼群/气泡/斜光柱 + **四机位验收**。
