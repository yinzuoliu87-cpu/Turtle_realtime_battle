## 3. 调查结论（带行号）

### 3.1 文案数据与渲染管线

**字段命名不统一**：技能是 `brief`/`detail`，被动是 `brief`/**`desc`**（不叫 detail）。导致 4 个文件里散落 6 处 fallback 链，口径还不一致（有的 detail→brief，有的 brief→detail）。

**`scripts/engine/skill_text.gd`（219 行）能力**：
- 占位符 `{X:expr}`，X ∈ `N`物理红 / `M`法术蓝 / `T`真伤白 / `P`穿透 / `S`护盾 / `H`治疗绿 / `B`增益 / `D`防御黄；`{expr}` 计算但不上色
- `eval_expr`（`:53-66`）用 Godot `Expression` 求值后 `roundi`；**解析失败静默返回原串**
- `build_vars`（`:70-141`）约 100 个变量；实测 pets.json 只用到 17 个，**全部有定义**
- 26 条关键词自动上色规则（`:35-49`）
- 三个出口：`render_html` / `render_bbcode`（给 RichTextLabel）/ `render_plain`（**整段剥成纯文本**）

**装备文案完全不走这条管线** —— 59 件的 `effectDesc1` 里**零处占位符、零处 HTML**，纯手写斜杠三元组。

### 3.2 五个展示区块现状

| 区块 | 位置 | 控件 | 显示 | 上色 |
|---|---|---|---|---|
| 选龟·被动/技能 tooltip 与弹窗 | `TeamSelectScene.gd:1605/1708/1797`、`SkillTipButton.gd:27,36` | `Label` | `brief` | **❌ 全灰** |
| 战斗·信息面板 被动段 | `RealtimeBattle3DScene.gd:23442` | `Label` | **`desc`**（详细） | ❌ |
| 战斗·信息面板 技能段 | `:23615` | `Label` | **`brief`**（缩略） | ❌ |
| 图鉴 | `CodexScene.gd:1088/1196/1214` | **`RichTextLabel`** | brief/detail/desc | **✅ 唯一有色** |
| 商店 / 背包 | `ShopScene.gd:364`、`InventoryScene.gd:567/631` | `Label`/系统 tooltip | `effectDesc1` 裸串 | ❌ |

**同一个信息面板里，被动段给详细、技能段给缩略 —— 口径自相矛盾。**

### 3.3 已证实的文案错误（实测差分）

| 龟 | brief | desc | 后果 |
|---|---|---|---|
| **天使龟** | 目标当前生命值 **11%** 魔法伤害 | **8%** | 选龟界面说 11%、战斗面板说 8% |
| **忍者龟** | **500** 码内可冲 / 固定 **450** 码冲刺斩 | **290** 码 / **300** 码 | 同上，差得很远 |
| 钻石龟 | "全队护甲魔抗加成效果额外放大 +50%" | "登场时全队护甲魔抗各 +50%" | **是两种机制**，非数字差 |

> ★调查报告曾称"28 条 skillPool.detail 缺失"，**实测 112 格一条不缺**（它把双形态龟的其它数组算进来了）。这也解释了为什么门禁一直是绿的。

### 3.4 交互冲突（测试人员反馈）

**所有输入都在一个函数里**：`RealtimeBattle3DScene.gd:20416 _unhandled_input`（平移/滚轮缩放/捏合缩放/点单位）。全场景**零处** `set_input_as_handled()`。

| # | 冲突 | 证据 | 现象 |
|---|---|---|---|
| **A** | 面板在 **press** 开/关，而拖动阈值在 **release** 判 | `:20501` press 就 `_show_unit_info_panel`；`:20456` 的 `_pan_moved` gate 在 release | 从单位附近起手拖视角 → 面板瞬间弹出；从空白起手 → 面板被关。**单位命中半径 64px**，几乎没有安全起手点 |
| **B** | `_pan_active` 会永久卡 true | `:20459` 只看布尔**不看 `button_mask`**；release 落在任何 STOP 控件上就永远收不到 | **不按键、纯移鼠标镜头一直跑**。同文件 `:2612` 的放置拖拽做了这层兜底，平移漏了 |
| **C** | 手机平移速度 **2 倍** | `emulate_mouse_from_touch` 默认开，单指拖同时产生 `ScreenDrag`(`:20465`) 与模拟 `MouseMotion`(`:20459`)，各跑一次 | `InventoryScene.gd:639` 记过这个坑，平移又踩 |
| **D** | 捏合抬指后手势被误接管 | pinch 分支从不置 `_pan_moved`；抬指后 `_pan_active` 仍 true、`_pan_from` 还是最初落指点 | 剩下那根手指直接接管平移 |

**「点暂停后拖不动」—— 两个独立根因**：

1. **场景根节点没设 `process_mode`**（`scenes/RealtimeBattle3D.tscn:5` 无该属性）→ 继承默认 `PAUSABLE` → `get_tree().paused = true` 后 Godot **不再调用它的 `_unhandled_input`**。所有交互都在那一个函数里。
2. **暂停暗幕全屏 + `MOUSE_FILTER_STOP` + `PROCESS_MODE_ALWAYS`**（`:1336-1340`）→ 就算修好第 1 条，事件也会被它全吃掉。

**★ 门禁是假的**：`tests/verify_cam_pan.gd:126` 断言「超阈值的拖动会 return（不误开/误关详情面板）」，实现却只是 `ui.contains("_pan_moved")` —— **只要函数体里出现这七个字符就 PASS**。全文件 **0 处**构造过 `InputEvent`，也没验过暂停。描述完全正确，断言形同虚设。

## 4. 方案

### 需求1 · 阅读体验（按区块）

**总原则（照 Riot 口径，用户拍板）**：缩略给算好的实时值，详细才展开比率。

| 阶段 | 内容 |
|---|---|
| **1-a 统一取值口径** | 新增 `SkillText.get_brief(d)` / `get_detail(d)`，内部做 `detail→desc→brief` 三级 fallback，把 6 处散落的 fallback 链收口。**不重命名 `passive.desc`** —— 该字段名被 3 个工具硬编码，重命名收益低于成本 |
| **1-b 统一色表** | 现有**三张互不相干的色表**：`skill_text.gd:24 VAL_HEX`(17项) / `dmg_stats_panel.gd:12-16 COL_*`(5项) / `visual_constants.gd FLOAT_STYLE`。物理红就有 `#ff6b6b` 和 `#ff4444` 两个值。合并成一张 |
| **1-c 上色铺开** | `SkillTipButton.gd:27,36`（选龟 tooltip，**性价比最高**）、`TeamSelectScene.gd:681` 弹窗、`RealtimeBattle3DScene.gd:23695 _add_body_text` 全部 `Label → RichTextLabel(bbcode)`，渲染改 `render_bbcode`。★战斗面板 `:23098` 有每帧比对 `slb.text` 的节流，换控件后要改比对源 |
| **1-d 装备文案按星级高亮**（用户拍板：**不改数据格式**） | 渲染时把玩家当前星级那一档加粗上色、其余两档变暗。零风险 —— 不碰 `tooltip_number_audit.py` 依赖的三元组正则 |
| **1-e 修文案错误** | 天使龟 11%↔8%、忍者龟 500/450↔290/300、钻石龟机制描述二选一。**改前要先确认代码里的真值是多少**（代码是最终事实源） |
| **1-f 两级描述交互** | 战斗信息面板技能段改成默认 brief + 「展开详细」；选龟界面同理。统一口径（现在被动给详细、技能给缩略是矛盾的） |
| **1-g 背包→商店** | `InventoryScene` 加「🛒 商店」按钮；顺带给 `ShopScene` 补上开放条件校验（现在能绕过） |
| **1-h 新手引导接线** | `TutorialGuide.gd` 已完整，只需：新增 `data/tutorial-steps.json` + 在战斗场/选龟/商店 `preload().new()` + `start(steps, cb)` + 关键动作处 `notify(event)`；主菜单 `_on_tutorial` 改成真正走引导；战斗场右上加第 4 个 `❓` 按钮（x=1028 顺位） |

### 需求2 · 版本号（我的方案，待你点头）

**拳头的做法**：`赛季.补丁` 两段（如 `14.1`、`14.2`），约两周一个补丁，每个补丁配一篇 patch notes。**不是 semver** —— 因为游戏没有"API 兼容性"概念，玩家关心的是"这是第几次更新、改了什么"。

**建议本项目采用**：

```
version = "0.9.3"          # 大版本.功能版本.改动序号
```

| 位 | 什么时候 +1 |
|---|---|
| 第 1 位 | 大阶段（0=测试期，1=正式发布） |
| 第 2 位 | 加了新玩法/新系统（如训龟大师上线） |
| **第 3 位** | **每次有玩家可感知的改动**（=你说的"每一次改动为一个小版本"） |

配套：
- `project.godot` 加 `application/config/version="0.9.3"`（Godot 标准字段，导出时自动带进包）
- 主菜单右下角显示 `v0.9.3`，**测试者报 bug 时能说清是哪个版本**（这是版本号最大的实际价值）
- 新增 `CHANGELOG.md`，每次改动一行，格式照 patch notes：`## 0.9.3 — 2026-07-22` + 分类列表
- 出包时 CI 自动把版本号写进 artifact 名
- **门禁**：断言 `project.godot` 的版本号 == `CHANGELOG.md` 最新条目的版本号（防止改了代码忘了记账）

### 需求3 · 训龟大师（用户拍板：**先做本体，法术后插**）

**本体规格**（用户给定）：
- 500 HP / 1 ATK / 0 护甲 / 0 魔抗
- **不会被主动索敌**，但**吃 AOE 伤害**
- 被动：**受到的所有类型伤害降为 1**（含真实伤害）
- 射程 2000，普攻 = 扔石头，1 点物理伤害
- 双方各一个：己方玩家控制，对面人机控制
- 需要几个形象供选择

**技术落点**（基于现有代码）：

| 项 | 怎么做 | 依据 |
|---|---|---|
| 不被主动索敌 | `_nearest_enemy` 已有 `_egg_fence` 跳过的先例（`:4525`），照做加 `is_trainer` 跳过 | 现成模式 |
| 吃 AOE | `_enemies_of` **不跳过**它即可（AOE 走这条） | 与索敌天然分离，正好符合需求 |
| 伤害降为 1 | 在 `_mitigate_incoming` 末尾加：`if u.is_trainer: return 1.0`。**放最后**，这样连真伤也覆盖 | 我们刚建的公共减伤函数，两条路径共用 |
| 存活/胜负 | 训龟大师**不应计入团灭判定**（否则打不死它就永远不结束）→ `_dl_side_alive` / `_check_end` 跳过，照 `_isEgg` 的先例 | `:2744` 已有同类分支 |
| 形象选择 | 存 `GameState`，选龟界面加一格 | — |

**待设计（下一轮）**：5 选 1 法术、手动拖放交互、冷却/资源。
**我的初步想法**（不是方案，供你后续拍板）：Botworld 的法术是"局内手动释放、有冷却、靠拖放指定落点"。本项目已有 `_pending_shots` 延迟执行、`_skill_ring` 落点指示、`_edit_drag_unit` 拖拽三套现成积木，接得上。

### 需求4 · 交互冲突（用户拍板：**根节点那条也改，但先摸清 tick 挂点**）

按风险从低到高：

| 序 | 改动 | 风险 |
|---|---|---|
| 1 | `:20459` 加 `and (event.button_mask & MOUSE_BUTTON_MASK_LEFT) != 0`（对齐 `:2612`） | 极低，止血 |
| 2 | `:20501` 开/关面板从 **press** 挪到 **release** 且以 `not _pan_moved` 为前提 | 低 |

| 5 | 移动端双事件流二选一（`ScreenDrag` 或模拟鼠标，不能都要） | 中 |

| 7 | **根节点 `process_mode = ALWAYS`** + 战斗 tick 逐个加 `paused` 早退 | **高 —— 必须先逐个核对 tick 挂在根节点还是子节点，否则暂停会失效** |

## 5. 已知风险与未决点

### 高风险

| # | 风险 | 说明 |
|---|---|---|
| **R1** | **`tools/tooltip_number_audit.py` 会静默失效** | 它用正则从 `effectDesc1` 抠 `0.6/0.75/1.0` 三元组。**格式一变就抠不出来 → total 归零 → 打印 ALL OK 但什么都没查**。用户已拍板"不改数据格式"，风险规避；但**仍建议给它加"抠到的三元组不得少于 N 条"的下限断言**，因为它现在就是静默的 |
| **R2** | **根节点改 ALWAYS 会让暂停失效** | 战斗 tick 若挂在根节点 `_process`，改完暂停就不停了。**实施前必须逐个核对挂点** |
| **R3** | `tools/data_integrity.py:62-65` 检查 `<span>` 开闭配对 | 若把 HTML span 换成 BBCode，**这条检查变成永远通过的空检查**（静默失效） |
| **R4** | `skill_text.gd` 公式写错**静默显示原文** | `eval_expr` 解析失败原样返回。历史上已发生过（`:23228` 注释记载玩家看到 `{N:0.7*ATK}`）。**建议新增测试：遍历 pets.json 所有模板，断言渲染后不含 `{`** |

### 轮次 A · 需求4 交互冲突（2026-07-22 完成，门禁 51/51）

**方案表里 7 条的实际落点：**

| 序 | 方案原文 | 实际结果 |
|---|---|---|
| 1 | 加 `button_mask` 兜底 | ✅ 已加。反向验证：拆掉后"空手移鼠标"平移 9.0m |
| 2 | 开/关面板 press → release | ✅ 已改。连带修好 `verify_info_panel` 的切片断言（见下） |
| 3 | `_toggle_pause` 清脏态 | ✅ 已加，4 个字段各一条断言 |
| 4 | 暗幕 STOP → IGNORE | ✅ 已改，**但它不是根因**（原方案判断错误，见下） |
| 5 | 移动端双事件流二选一 | ✅ `_touch_seen`：收到真触屏后忽略模拟鼠标 |
| 6 | 捏合抬指需重新落指 | ✅ 抬指后 `_pan_from` 重置到剩余手指当前位置 |
| 7 | 根节点 `process_mode = ALWAYS` | ❌ **没这么做**，改用 `_CamInputRelay`（见下） |

**与方案的三处偏差，都是被实测推翻的：**

1. **R2 的担心成立，但解法换了。** 全场 `ALWAYS` 会让 `_process` 里的战斗 tick 在暂停中照跑
   = 取消暂停。改为：把相机输入抽成 `_cam_handle_input(event) -> bool`，另挂一个
   `PROCESS_MODE_ALWAYS` 的空壳节点 `_CamInputRelay`，**只在 `paused` 时**转发。
   主场景保持 PAUSABLE，暂停语义完全不变。原计划的"逐个 tick 加 paused 早退"整条免了。

2. **方案里"暗幕 STOP 是第二个根因"是错的。** 2×2 探针实测：

   | process_mode 修 | 暗幕 filter 修 | 暂停中事件到达 `_unhandled_input` |
   |---|---|---|
   | ✗ | ✗ | false |
   | ✓ | ✗ | **true** |
   | ✗ | ✓ | false |
   | ✓ | ✓ | true |

   **单改暗幕无效，单改 process_mode 就够** —— 真根因只有一个：主场景 `INHERIT` 跟着
   root 的 `PAUSABLE`，暂停时 `can_process()=false`，`_unhandled_input` 一次都不被调用。
   代码里那条说"第二个根因"的注释已就地改正（`_build_pause_panel`）。

3. **第一版探针给出过相反的错结论。** 它把战斗场 `add_child` 到探针节点（`ALWAYS`）下，
   `INHERIT` 继承成 `ALWAYS` → 测出"暂停能拖"。**测进程模式必须挂 `get_tree().root`**，
   这条已写进 `verify_cam_pan.gd` 的注释和测试本身。

**测试改造：`verify_cam_pan` 从查源码字符串改为真发事件序列**（5 组 14 条断言）。
理由：这批全是时序 bug，静态查串一条也抓不到；且相机代码一搬家，两条写死
`_unhandled_input` 的旧断言就红了，而功能完全正常。

**反向验证（逐条破坏 → 必须红）：**

| 破坏点 | 结果 |
|---|---|
| 拆掉 `_CamInputRelay` 挂载 | 🔴 暂停中拖动量 `2.52 → 0.00` |
| 拆掉 `not _touch_seen` | 🔴 模拟鼠标重复平移 `1.26 → 2.52`（正好 2 倍） |
| 拆掉 `button_mask` 判断 | 🔴 空手移鼠标平移了 9.00 |
| 拆掉捏合抬指重置 | 🔴 `_pan_from` 停在最初落指点 |
| 拆掉暂停清脏态 | 🔴 `_pan_active` / `_pan_moved` 两条 |

**过程中抓到的三个"假绿灯"，都写进了测试注释防复发：**

- **饱和假象**：第一版用 4×40px 拖动，`push_input` 会按内容缩放放大 20×，
  一帧吃 16.8m 被 clamp 死在 `PAN_LIMIT=9.0`。于是"暂停 9.0 vs 平时 9.0"看着漂亮，
  其实是两个饱和值在比 —— 拆掉去重照样绿。已加 `_assert_not_saturated`。
- **空检查**：测"模拟鼠标不重复平移"时没推送系统同时发出的**模拟鼠标按下**，
  `_pan_active` 恒为 false，条件根本不成立 → 断言恒真。已补按下事件 + 前置断言。
- **切片窗口两头都是坑**（`verify_info_panel`）：`substr(ui,4000)` 再 `substr(pl,700)`
  实际只剩 255 字符 → 假 FAIL（差点据此去"修"一个没坏的功能）；放宽成 `substr(pl)`
  取到函数尾 → 函数内另外 2 处 `_close_info_panel()` 顶包 → 断言变哑。
  最终按**缩进**精确切分支，并加一条"切片没漏到隔壁分支"的自检。

**未做（留给后续轮次）：** 方案表第 4 条提到的"只覆盖按钮区的 STOP 控件防误点" ——
暗幕改 IGNORE 后按钮自身仍是 STOP，实测无误穿，暂不加多余控件。

### 9.2 推荐方案：**按住他本人拖动**（PC 与移动端同一套）

按下时先做一次命中判定（复用 `_edit_unit_at_screen`，就是详情面板用的那套 unproject 命中，半径 64px）：

- 命中的是**我方训龟大师** → 进入「拖他」模式：本次手势**不平移镜头、不开面板**
- 没命中 → 完全维持现状

**为什么这个方案最稳**：它和现有手势的区分点在**按下那一刻的命中判定**，而不是靠时间、长按、或第二根手指 —— 而"按下时判定命中"这条路径已经在详情面板上跑了一整天，是验证过的。鼠标与触屏走同一段代码，PC/移动端天然一致，不需要两套逻辑。

**PC 额外给一条**：右键点地面 = 命令他走过去。零冲突（右键正式局里空着），适合"我正看着别处，让他挪个位"。

**移动端明确不做虚拟摇杆**：摇杆要常驻占屏，而这是自走棋，移动是**低频**操作 —— 为低频操作长期占掉一块屏幕不划算。

### 9.4 验收标准（怎么证明这个检查会 FAIL）

事件序列测试（照 `verify_cam_pan` 的写法，真发 `InputEvent`）：

1. 在训龟大师身上按下并拖动 → 他的 `pos` 改变，且**镜头 `_cam_pan` 不变**
2. 在空白处按下并拖动 → **镜头动、他不动**（反向对照，防止"按住哪都能拖他"）
3. 在他身上按下但位移 < 阈值就抬起 → 开他的详情面板（不算拖动）
4. 破坏验证：把命中判定去掉 → 第 1 条与第 2 条必须**同时**红
